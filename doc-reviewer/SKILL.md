# 文档评审

本文档定义了技术文档评审的标准操作规范和检查清单，旨在确保项目文档的高质量与一致性。

---

## 1. 评审流程

标准评审流程包含从目标确认到结果反馈的完整闭环，以确保评审过程系统、透明且高效。

1. **确认目标**：明确用户需要评审的文件或文本段落。
2. **读取内容**：读取目标文档的全部内容。
3. **执行评审**：逐项对照下方"评审规则"进行检查。
4. **反馈结果**：
   - 如发现问题，清晰列出并指明具体行号或章节。
   - 提供具体的修改建议。
   - 若文档符合所有标准，明确告知用户已通过评审。
5. **自动修正（可选）**：如用户要求，直接应用修复，修复后总结所做的更改。
   - **静默修正**：对于基础格式问题（如中英文空格、全半角标点等），可直接进行修正。
   - **需确认修正**：对于涉及内容重构、大规模术语替换或有歧义的代码修改，必须先列出建议并等待用户确认。

## 2. 评审规则

评审规则涵盖了文档的各个维度，作为执行评审时的主要判断依据。

### 2.1 结构与格式

本节规定了文档在排版结构和基础格式上的要求，包括标题层级、标点使用及中英文排版等。

- **章节编号**：文章章节需逐级编号（1., 1.1, 1.1.1），但 Markdown 标题本身（`#`, `##`, `###`）**不带编号**，编号仅体现在标题文字中。
  - 正确：`## 1. 核心概念`
  - 错误：`## 1. 1. 核心概念`（重复编号）
- **标题语言**：标题只使用一种语言，**不使用双语标题**。
  - 正确：`## 核心架构设计`
  - 错误：`## 核心架构设计 (Core Architecture Design)`
- **概述段落**：每个章节标题及子小节标题后，必须紧跟一段概述文字，不得直接跳到子标题或列表。
  - 正确：

    ```text
    ## 3. 核心架构设计

    本章将详细解析系统的核心架构，包括分布式存储池和双阶段调度机制。

    ### 3.1 分布式存储池
    ```

  - 错误：

    ```text
    ## 3. 核心架构设计

    ### 3.1 分布式存储池
    ```

  - 特别注意：**如果父章节标题下紧接着子章节标题（如 `## 1.` 后紧接 `### 1.1`），必须在两者之间增加一段过渡性的介绍文字。**

- **章节分隔**：主要顶级章节之间使用 `---` 水平线分隔。
- **中英文空格**：中英文内容之间必须有空格。
  - 正确：`使用 Python 编写`、`基于 RDMA 的传输`
  - 错误：`使用Python编写`、`基于RDMA的传输`
- **中文与数字空格**：中文与数字之间也须有空格。
  - 正确：`延迟低于 50 μs`
  - 错误：`延迟低于50μs`
- **标点符号**：中文文档主要使用全角标点（，。：；？！），但数字、英文单词、代码片段周围使用半角标点。
  - 正确：`支持 vLLM、TGI 等框架。`
  - 错误：`支持vLLM,TGI等框架.`
- **专有名词**：严格遵循官方大小写规范。
  - 正确：`vLLM`, `Redis`, `MySQL`, `iOS`
  - 错误：`vllm`, `redis`, `mysql`, `ios`
- **表格格式**：Markdown 表格中**不使用** `<br>` 等 HTML 标签。表格内容需对齐，列宽合理。
- **列表格式**：使用加粗关键词 + 冒号的格式组织定义列表。列表项若为完整句子，结尾需加标点；若为短语，可不加。
  - 正确：`- **L1 极速内存层**：GPU 显存，保存活跃数据。`
- **图片引用**：使用相对路径，路径中的空格需编码为 `%20`。

### 2.2 内容质量

本节重点关注文档的技术准确性、术语一致性以及表达的专业度，确保内容传达无误。

- **安全与脱敏**：检查代码块或配置示例中是否包含真实的 API Key、密码、内部 IP 或私人敏感信息，要求使用如 `YOUR_API_KEY` 等占位符。
- **缩写规范**：专业术语或缩写在文档中首次出现时，应提供全称或简要解释（如 `LLM (Large Language Model)`）。
- **链接有效性**：确保文档中的所有内部跳转链接（锚点）和外部超链接格式正确，逻辑上可达。
- **准确性**：技术描述必须准确无误，公式、数据、术语均需正确。
- **一致性**：全文术语和描述前后保持一致。选定一种表述后通篇统一。
  - 例：始终用 "KV Cache"，不混用 "KV 缓存"、"键值缓存"。
- **数据来源**：所有引用的数据、实验结果须说明来源或标注参考文献。
- **文风**：语言风格严谨、专业，避免口语化表述和情绪化用语。
- **参考文献**：
  - **引用标记**：正文中使用方括号编号引用（如 `[1]`, `[2]`）。
  - **格式规范**：推荐采用 IEEE 格式。
    - 在线资源：`[编号] Author, "Title," *Source*, Year. [Online]. Available: URL`
    - 标点位置：标题后的逗号应置于引号内（如 `"Title,"`）。
    - 来源斜体：出版方或网站名称应使用斜体（如 `*OpenAI*`）。
  - **无需介绍**：参考文献章节不需要开头介绍段落，直接列出文献即可。
- **叙述方式**：使用直接叙述的方式表达内容，避免使用"更新："、"现状："等非文档正文风格的元数据前缀。
- **人称代词**：中文文档中应避免使用“你”，根据语境选择合适的代词：
  - **泛指受众/读者群体**：使用“大家”。例如：“为大家提供实战指南”。
  - **特指读者/开发者自己的事物或行为**：使用“我们”。例如：“我们的软件必须支持这种工作流”、“重新审视我们的 API 设计”。
  - **可省略或客观化**：当上下文清晰时，可省略人称或改用客观表述。例如：“Agent 理解你能力的” → “Agent 理解软件能力的”。

### 2.3 代码质量

本节规范了文档中内嵌代码块的书写标准，强调代码的可读性、性能及注释完整性。

- **代码注释**：代码块必须包含注释说明关键逻辑。
- **语言标注**：代码块须标注语言类型（如 ` ```python `）。
- **代码风格**：遵循良好编程风格，兼顾**可读性**、**性能优化**与**可维护性**。

### 2.4 可视化元素

本节说明了文档中图表、公式等非纯文本元素的正确表达和渲染规范。

- **Mermaid 图表**：优先使用 Mermaid 绘制架构图和流程图，避免使用外部工具生成的不可编辑图片。
- **ASCII 图**：对于简单的示意图，可使用 ASCII 图快速表达，例如：

  ```text
  +--------+     +--------+     +--------+
  | Client | --> | Server | --> |  DB    |
  +--------+     +--------+     +--------+
  ```

- **LaTeX 公式**：
  - **基础语法**：数学公式使用 `$...$`（行内）或 `$$...$$`（块级）LaTeX 语法。
  - **公式前后空格**：行内公式 `$...$` 与前后文本之间必须有空格，以确保 GitHub 等平台正确渲染。**正确示例**：`长度： $P$ tokens`；**错误示例**：`长度：$P$ tokens`。
  - **文本与下标**：公式中的描述性文本或多字符下标使用 `\text{...}`（如 `\text{Memory}_{KV}`）。
  - **代码变量**：涉及代码变量名时，使用 `\texttt{...}` 包裹以保持等宽字体风格（如 `\texttt{block\_size}`）。
  - **运算符与符号**：乘法使用 `\times` 而非 `*`；近似关系使用 `\approx`；千分位分隔符使用 `{,}`（如 `57{,}344`）。
  - **单位标注**：数值与单位之间使用 `\,` 分隔，单位使用正体 `\text{...}`（如 `1.12\,\text{GiB}`）。
- **对比表格**：当需要对比多个方案或特性时，使用 Markdown 表格清晰呈现。

## 3. 评审输出格式

规范化的输出格式能够帮助用户快速定位问题并理解修改建议。

按以下格式报告问题：

```text
## 评审结果

### 发现的问题

1. **[类别] 行 XX**：问题描述。
   - 建议：具体修改建议。

2. **[类别] 第 N 节**：问题描述。
   - 建议：具体修改建议。

### 总结

共发现 X 个问题（格式 X 个，内容 X 个，代码 X 个）。
```

问题类别包括：`格式`、`内容`、`代码`、`可视化`。

## 4. 示例

以下示例展示了文档评审过程中的标准交互流程与典型反馈。

**用户**："帮我评审一下 `architecture.md` 文档。"

**Agent**：

1. 读取 `architecture.md`。
2. 逐项检查评审规则。
3. 回复：
   > 我已完成对 `architecture.md` 的评审，发现以下问题：
   >
   > 1. **[格式] 标题 "系统设计" 下**：缺少开头的概述段落。
   > 2. **[格式] 行 15**：`使用Redis作为缓存` 缺少中英文空格，建议修改为 `使用 Redis 作为缓存`。
   > 3. **[内容] 行 32**：术语不一致，前文使用 "KV Cache"，此处使用 "键值缓存"，建议统一。
   > 4. **[代码] 行 50**：Python 代码块缺少注释。
   > 5. **[格式] 行 78**：表格中使用了 `<br>` HTML 标签，建议拆分为多行或使用列表。
   >
   > 共发现 5 个问题（格式 3 个，内容 1 个，代码 1 个）。
   > 您是否希望我自动修复这些问题？
