# 角色协同与工作流详述

本文件详细定义了基于变更 (Change) 的完整生命周期中，三个核心角色在不同阶段的具体职责与核心动作。

为了保证开发的规范性，我们将工作流拆分为三个核心角色，他们在 Change 的生命周期中各司其职：

## 1. 架构师 (Architect) - 意图对齐、规格生成与评审归档阶段

本小节规定了架构师在 Change 生命周期的起点和终点所承担的职责。

**职责**：在起点进行领域建模与规格定义，确保需求清晰；在终点进行代码评审，确保实现与规格一致。
**核心动作**：

- **起点：初始化与创建**：执行 `openspec init --tools none` (首次) ，并执行 `openspec new change <change-name>` 开启新的里程碑/变更。
- **起点：输出规格产物**：(可参考本技能内置模版 `.trae/skills/openspec/templates/v1-mvp/` 作为强约束范例)
  - `proposal.md`：明确业务目标与范围，必须包含 `## Why` 和 `## What Changes` 章节。
  - `design.md`：确定模块边界与数据流向，划分领域层、服务层与基础设施层。
  - `specs/<capability>/spec.md`：定义能力规范。必须使用 `## ADDED Requirements` 等 Delta Header ，包含 `### Requirement: <标题>` 和 Gherkin 格式的 `#### Scenario: <标题>` 。
- **起点：规范校验**：执行 `openspec validate <change-name>` ，确保格式 100% 正确，完善并检查 Spec 质量。
- **终点：代码评审与归档**：在 Developer 和 QA 完成工作后，严格对照 Spec 逐条评审生成的代码，确保不偏离规格。评审通过后，响应 `/opsx:archive` 或执行 `openspec archive <change-name>` 将变更合并入主基线。

## 2. 开发工程师 (Developer) - 规范驱动的实现生成阶段

本小节指导开发工程师如何严格依据规范文档与测试用例进行代码落地，建立追踪矩阵，确保实现质量。

**职责**：严格按照 Spec (模型、接口定义、异常处理) 与 QA 提供的测试基准实现业务逻辑，拒绝偏离规范的自由发挥。
**核心动作**：

- **实现生成**：响应 `/opsx:apply` ，进行代码落地。在生成代码前，必须先通读 `proposal.md` 和 `design.md` 获取全局上下文。
- **分层映射与实现**：
  - **领域层 (Domain)**：精确映射 `domain/spec.md` 中的数据模型（保证字段、类型、非空约束一致）。
  - **服务层 (Service)**：实现核心业务规则与用例编排，确保模块间边界清晰。
  - **接口层 (Interface)**：精确映射 `api/spec.md` 中的 HTTP 方法、路由、状态码及响应结构。
  - **异常处理**：严格遵循 Spec 中定义的错误码规范（如 400 Bad Request, 409 Conflict），在接口层做好映射。
- **追踪矩阵 (Traceability Matrix)**：编写代码时，必须在注释中引用对应的 Spec 章节 (例如 `// 对应 Spec: POST /api/orders`) ，建立 Spec ↔ 代码 ↔ 测试的强追溯映射。
- **自证闭环**：在交付前，必须运行 QA 编写的验证脚本（包含冒烟测试和单元测试），直至所有自动化测试绿灯通过，确保逻辑无漏洞。

## 3. 测试工程师 (QA) - 自动化验证闭环阶段

本小节明确了 QA 角色如何将规范文档转化为可执行的测试基准，实现测试驱动与自证清白。

**职责**：作为规范的“契约守护者”，基于 Architect 产出的 Spec 设计并编写自动化测试用例，从正向流程、边界条件和异常处理等全方位验证 Developer 的代码，最终交付“代码可工作”的客观证据。
**核心动作**：

- **场景映射与用例设计**：深度参考 Spec 中的 `Scenario` (Gherkin 格式的 Given/When/Then) ，在测试目录 (如 `__tests__/`) 编写对应的自动化测试脚本。
  - **集成测试 / 冒烟测试**：模拟真实用户路径，验证跨模块协作（如加购 -> 下单 -> 扣库存的业务闭环）。
  - **单元测试 / 异常测试**：针对纯函数逻辑、边界条件（如库存不足、重复下单等）编写细粒度用例，确保异常状态码映射正确。
- **测试驱动 (TDD 协同)**：在 Developer 编码前或并行建立性能基线 (如 p99 延迟) 与核心逻辑的测试骨架，为开发提供明确的“红绿验收标准”。
- **测试执行与反馈**：在 Developer 编写代码后，运行完整的测试套件。如果测试失败（红灯），需提供清晰的错误反馈，驱动 Developer 修复逻辑，直至系统行为严格符合 Spec 预期（全绿通过）。
